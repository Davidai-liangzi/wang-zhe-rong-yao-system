# reflection.md - AI Usage Reflection

---

## 1. Which AI tools or models did you use?
Deepseek-V4-Pro.

## 2. Which prompt was the most useful? Why?
Prompt 16. 
It solved six problems. 
Firstly, it added Identifiable interface, giving an id field to every class, linking heroes to equipment.
Secondly, it expanded data to five players and per team with three heroes each, and adding match record management.
So it has released the stress of the project programming.

## 3. Which AI-generated suggestion was wrong, incomplete, or misleading?
The findPlayerByName method (Prompt 06) generated by AI was logically correct, but when integrated into Main.java, the AI didn't account for a Scanner issue: `sc.nextInt()` leaves a newline character in the input buffer, which causes the next `sc.nextLine()` to be skipped. This meant the user couldn't actually type a player name to search — the input was silently bypassed.

## 4. How did you check whether AI-generated code was correct?
I used four methods: 
1.Compiled every AI-generated Java file immediately 
after generation — if `javac` reported errors, I fixed them before 
committing. 
2.Wrote FullTest.java (later renamed TestRunner.java) with 113 automated assertions covering 
all 11 modules, ran it multiple times after major changes, and verified 
100% pass rate. 
3.Interactively tested features through the console 
  menu — for example, searching for "p1_fly" to verify player lookup, 
  adding/deleting a test player to verify CRUD, and checking that data 
  persisted after restart.
4.In the final testing phase, wrote BoundaryTest.java with 180 JUnit5 
  parameterized boundary test cases covering all 17 boundary-critical 
  methods, and executed mutation testing with 20 mutants — all 20 
  killed (100% mutation score), proving the test suite catches injected 
  faults.

## 5. What bugs did you fix yourself instead of asking AI to fix?
When I added the Identifiable interface, javac reported that Player 
and Admin didn't implement getName(). The AI had added the interface 
declaration but forgot that Person used getUsername() instead of 
getName(). I fixed this by adding `public String getName() { return 
username; }` to Person — one line in the abstract parent solved 
both errors.

## 6. What Java concept did you understand better after using AI?
Before this project, I had only studied up to Chapter 11 of my Java 
textbook (inheritance and polymorphism). Through AI-assisted development, 
I learned several concepts beyond the textbook:
1.Abstract classes — Person was abstract, which I didn't know was 
   possible before. The keyword "abstract" prevents creating a plain 
   "Person" directly.
2.Interfaces — Identifiable taught me how to define a contract that 
   multiple unrelated classes (Person, Hero, Equipment) can share.
3.Serialization — ObjectOutputStream/ObjectInputStream to save and 
   load the entire game state to data.ser. I had never written code 
   that persists beyond program exit before.
4.Enums — HeroRole, EquipmentType, Role. Using enums instead of 
   String constants made the code safer (compile-time checks).
5.Stream API — filter().sorted().limit() in match history was much 
   cleaner than writing manual loops and counters.

## 7. What Java concept are you still unsure about?
Honestly, since most of the code was AI-generated, many concepts 
are still fuzzy to me. Two specific ones:

1. Stream API with lambda expressions — in showMatchHistory(), the 
   code uses `stream().filter(m -> ...).sorted().limit(5)`. I can 
   read what it does (filter then sort then take 5), but I don't 
   fully understand how the lambda `m -> ...` captures variables from 
   outside, or why we use `::` syntax for method references.

2. Serialization — I know `implements Serializable` saves objects to 
   files, and I saw that changing the class structure breaks old saves. 
   But I don't understand why `serialVersionUID` is needed, or what 
   happens internally when ObjectOutputStream writes an object graph 
   with circular references (e.g., Player → Team → Player).

I plan to revisit these topics when my course reaches the relevant 
chapters (Stream API is Ch20, serialization is Ch17).

## 8. Did AI make the project easier, harder, or both? Explain.
Both. AI made coding much faster — generating entire Java classes 
in seconds that would have taken me hours to write from scratch, 
especially since I've only studied up to Chapter 11. It also 
introduced me to concepts beyond the textbook (enums, interfaces, 
serialization) through working examples rather than theory.

However, AI also made things harder in two ways. First, debugging 
AI-generated code was sometimes frustrating — the Scanner newline 
bug took me a while to figure out because the code looked correct 
at first glance. Second, I sometimes didn't understand code that 
compiled and ran perfectly, which felt uncomfortable. Overall, AI 
made the project possible at my current skill level, but I spent 
significant time verifying and understanding rather than just copying.

## 10.5 Advanced AI Reflection

I compared two AI agent roles — **Architect Agent** and **Implementation Agent** — on the same problem: designing and implementing the equipment ranking feature.

The same underlying model (Deepseek-V4-Pro via Trae IDE) was used for both, but assigned different roles via system prompts.

**Architect Agent** (Prompt 09 context): When asked about the equipment ranking formula, the Architect focused on:

- Whether the weight distribution made sense (ATK×1.0, DEF×0.8, HP×0.6, price×-0.001)
- Documenting the rationale in comments so future readers understand the design intent
- Recommending that the formula explanation go into plan.md as documentation
- The output was a design specification — not executable code

**Implementation Agent** (Prompt 09): Given the same formula, the Implementation Agent focused on:

- Writing the showEquipmentRanking() method in Java
- Copying the equipment list with new ArrayList() to avoid mutating original data
- Using List.sort() with Comparator chaining for clean descending sort
- Formatting printf table output with proper column alignment

**Key Differences**:

| **Dimension** | **Architect Agent** | **Implementation Agent** |
|---------------|--------------------|---------------------------|
| Focus | Why this formula, what weights mean | How to code it correctly |
| Output type | Text explanation + design principles | Compilable Java code |
| What I learned | Design rationale behind scoring systems | Stream API, Comparable/Comparator, printf formatting |
| Mistakes found | Suggested price penalty should be ×0.01 not ×0.001 (I kept 0.001) | None — code compiled on first attempt |
| Usefulness for me| Helped me think about fairness of rankings | Helped me finish the feature quickly |

**What I learned**: The same AI model produces fundamentally different outputs depending on the role it's assigned. The Architect role helped me understand *what* to build and *why*; the Implementation role helped me build it. Neither role alone would have been sufficient — the Architect gave me direction but no code, while the Implementation would have generated code without explaining why the weights were chosen that way. Using both roles gave me both understanding and a working feature.
